# Admin Employees Module

This module lets a logged-in Super Admin create employee login accounts.

It creates both records inside one database transaction:

- `users` stores authentication data and employee profile fields.
- `admin_departments` stores the selected department assignment.
- `users.userCode` is generated by the backend as `EMP001`, `EMP002`, and so on.

## Endpoint

```http
POST /api/admin/employees
Authorization: Bearer <super_admin_token>
Content-Type: multipart/form-data
```

Compatibility route:

```http
POST /api/v1/admin/employees
```

Only `SUPER_ADMIN` users can call this route.

## Request Body

```json
{
  "firstName": "Ali",
  "lastName": "Khan",
  "email": "ali@company.com",
  "password": "Employee@12345",
  "phone": "+923001234567",
  "address": "Lahore",
  "departmentId": 1,
  "designation": "Officer"
}
```

Required fields:

- `firstName`
- `lastName`
- `email`
- `password`
- `departmentId`

Optional fields:

- `phone`
- `address`
- `photo`
- `designation`

If no photo is being uploaded, `application/json` still works. The response includes the generated `employeeCode`.

For `photo`, submit the request as `multipart/form-data` with a file field named `photo`. Supported types are JPG, PNG, and WebP. The backend stores the file under `backend/uploads/employees` and saves the public `/uploads/employees/...` path in `users.photo`.

`joiningDate` exists in `users` as a nullable database field, but this create-employee flow does not set it. It is reserved for the employee-side first-login flow.

## Flow

```text
POST /api/admin/employees
  -> authMiddleware
  -> allowRoles("SUPER_ADMIN")
  -> uploadEmployeePhoto
  -> Joi validation
  -> controller
  -> service
  -> repository
  -> users + admin_departments transaction
```

## Files

- `admin-employees.routes.js` mounts the protected route.
- `admin-employees.validation.js` validates request fields.
- `admin-employees.controller.js` handles HTTP response shape.
- `admin-employees.service.js` checks business rules and hashes the password.
- `admin-employees.repository.js` writes `users` and `admin_departments`.

## Database

Employee details now live on:

```text
User
```

The department assignment relation is:

```text
User.adminDepartments -> AdminDepartment.department
```

The migration `database/migrations/20260714000000_move_profiles_into_users/migration.sql` copies old profile-table data into `users` and stores department links in `admin_departments`.
