# Admin Employees Module

This module lets a logged-in Super Admin create employee login accounts and employee profile records.

It creates both records inside one database transaction:

- `users` stores authentication data.
- `employee_profiles` stores employee HR profile data.
- `employee_profiles.employee_code` is generated by the backend as `EMP001`, `EMP002`, and so on.

## Endpoint

```http
POST /api/admin/employees
Authorization: Bearer <super_admin_token>
Content-Type: multipart/form-data
```

Compatibility route:

```http
POST /api/v1/admin/employees
```

Only `SUPER_ADMIN` users can call this route.

## Request Body

```json
{
  "firstName": "Ali",
  "lastName": "Khan",
  "email": "ali@company.com",
  "password": "Employee@12345",
  "phone": "+923001234567",
  "address": "Lahore",
  "department": "Operations",
  "designation": "Officer"
}
```

Required fields:

- `firstName`
- `lastName`
- `email`
- `password`

Optional fields:

- `phone`
- `address`
- `photo`
- `department`
- `designation`

If no photo is being uploaded, `application/json` still works. The response includes the generated `employeeCode`.

For `photo`, submit the request as `multipart/form-data` with a file field named `photo`. Supported types are JPG, PNG, and WebP. The backend stores the file under `backend/uploads/employees` and saves the public `/uploads/employees/...` path in `employee_profiles.photo`.

`joiningDate` exists in `employee_profiles` as a nullable database field, but this create-employee flow does not set it. It is reserved for the employee-side first-login flow.

## Flow

```text
POST /api/admin/employees
  -> authMiddleware
  -> allowRoles("SUPER_ADMIN")
  -> uploadEmployeePhoto
  -> Joi validation
  -> controller
  -> service
  -> repository
  -> users + employee_profiles transaction
```

## Files

- `admin-employees.routes.js` mounts the protected route.
- `admin-employees.validation.js` validates request fields.
- `admin-employees.controller.js` handles HTTP response shape.
- `admin-employees.service.js` checks business rules and hashes the password.
- `admin-employees.repository.js` writes `users` and `employee_profiles`.

## Database

The Prisma relation is:

```text
User.employeeProfile
```

The table is:

```text
employee_profiles
```

The migration `database/migrations/20260713000000_use_employee_profiles/migration.sql` creates `employee_profiles`, copies compatible data from `employees`, then drops `employees`.

The migration `database/migrations/20260713010000_add_employee_profile_photo/migration.sql` adds the optional `photo` column.
